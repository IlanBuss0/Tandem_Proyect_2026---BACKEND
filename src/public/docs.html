Base de Datos

PostgreSQL — 60+ tablas agrupadas por dominio funcional. Sin ORM.

¿Por qué PostgreSQL sin ORM?

Las consultas del negocio tienen JOINs complejos: usuarios → pertenecientes → vinculos → permisos.

Un ORM como Sequelize o TypeORM vuelve esas queries más difíciles de leer y optimizar.

Con SQL crudo tenés control total de cada query.

Podés hacer FOR UPDATE en refresh tokens, window functions para puntos, y transacciones sin magia.

Las entidades (78 clases) son objetos planos que se arman desde el constructor del PG result. Sin decoradores, sin mapeo automático, sin sorpresas.

Conexión a la DB

El Pool

Usamos pg.Pool que mantiene N conexiones abiertas (default 5) y las reusa.

Cada request obtiene una conexión del pool, ejecuta la query y la devuelve.

SSL está forzado con rejectUnauthorized: false tanto en producción como en desarrollo.

src/db/BD.js

Transacciones

Para operaciones que tocan múltiples tablas (ej: crear chat + participantes).

El método transaction(callback) obtiene un cliente dedicado, hace BEGIN / COMMIT / ROLLBACK y libera el cliente al final.

src/db/BD.js (método transaction)

import pg from 'pg'; const { Pool } = pg; class BD {   constructor() {     this.pool = new Pool({       connectionString: envConfig.databaseUrl,       max: envConfig.databasePoolMax,       ssl: { rejectUnauthorized: false },     });   }   query(sql, params) {     return this.pool.query(sql, params);   }   transaction(callback) {     const client = await this.pool.connect();     try {       await client.query('BEGIN');       const result = await callback(client);       await client.query('COMMIT');       return result;     } catch (error) {       await client.query('ROLLBACK');       throw error;     } finally {       client.release();     }   } }

Autenticación

JWT custom HMAC-SHA256 + Refresh Token con rotación + CSRF double-submit cookie

¿Por qué JWT custom sin librerías?

Firmar un HMAC-SHA256 es trivial con el módulo crypto de Node.

No vale la pena agregar una dependencia como jsonwebtoken para algo tan simple.

Esto nos da control total sobre el formato y evita vulnerabilidades de librerías de terceros.

El access token expira en 15 minutos. Incluso si se filtra, el daño es limitado.

La rotación de refresh tokens permite detectar robos:

Cada vez que se usa un refresh token, se crea uno nuevo y se revoca el anterior.

Si alguien usa un token ya revocado, la API invalida toda la familia y fuerza un relogin.

¿Cómo funciona el login?

Paso a paso

  1. 1. El frontend envía POST /api/auth/login con { correo, contrasena }.
  2. 2. AuthController llama a AuthService.login().
  3. 3. AuthRepository busca el usuario por correo en la DB.
  4. 4. AuthService verifica la contraseña con argon2.verify().
    Si el hash es SHA256 legacy (versión anterior), hace auto-upgrade a Argon2id en ese mismo login.
  5. 5. Si las credenciales son válidas, llama a createSession().
  6. 6. createSession() hace tres cosas:
    • Firma un JWT con HMAC-SHA256 con { id, id_tipo_usuario } y expiración de 15 minutos.
    • Genera un refresh token aleatorio de 64 bytes en base64url.
    • Guarda el SHA256 del refresh token en la tabla refresh_tokens con un family_id UUID.
  7. 7. El controlador setea tres cookies en la respuesta:
    tandem_access_token (HttpOnly),
    tandem_refresh_token (HttpOnly),
    tandem_csrf_token (no HttpOnly, accesible por JS).
  8. 8. La respuesta JSON vuelve con { ok: true, usuario }.

¿Cómo se protege cada request?

authMiddleware

Se aplica a todas las rutas /api/* excepto auth y pictogramas.

  1. 1. Lee tandem_access_token de req.cookies.
  2. 2. Verifica la firma HMAC-SHA256 con timingSafeEqual.
  3. 3. Verifica que exp no haya expirado.
  4. 4. Inyecta req.user = payload.
  5. 5. Si algo falla, responde 401.

src/middlewares/auth.middleware.js

csrfMiddleware

Protege contra ataques CSRF. Se aplica después del authMiddleware.

  1. 1. GET, HEAD, OPTIONS pasan sin check.
  2. 2. Para otros métodos, lee x-csrf-token del header y tandem_csrf_token de la cookie.
  3. 3. Si no coinciden, responde 403.

Esto funciona porque la cookie tandem_csrf_token es accesible por JS pero un sitio malicioso no puede leer cookies de otro dominio.

El frontend envía el token como header personalizado, que un atacante no puede replicar.

Refresh Token — detección de robo

¿Cómo funciona la rotación?

Cada vez que el frontend llama a POST /api/auth/refresh:

  1. 1. Lee tandem_refresh_token de la cookie.
  2. 2. Busca el SHA256 del token en la tabla refresh_tokens.
  3. 3. Si el token fue revocado (tiene replaced_by_token_hash), significa que alguien lo está reutilizando.
    La API revoca toda la familia (todos los tokens con el mismo family_id) y responde 401 — el usuario debe reloguear.
  4. 4. Si el token es válido, crea uno nuevo, marca el anterior como revocado y responde con nuevas cookies.

Esto es el patrón Refresh Token Rotation de OWASP.

DB-->>A: { family_id, expires_at, replaced_by } alt Token revocado (replaced_by IS NOT NULL) A->>DB: UPDATE refresh_tokens SET revoked_at=NOW() WHERE family_id = ? Note over A: ¡Ataque detectado! Revocar toda la familia A-->>C: 401 → Forzar relogin else Token expirado A-->>C: 401 Expirado else Token válido A->>A: Firmar nuevo JWT + nuevo refresh token A->>DB: Revocar anterior + INSERT nuevo A->>C: Set-Cookie: nuevas cookies A-->>C: { ok: true } end

Chat Realtime

Socket.io con autenticación JWT, rooms, Redis adapter y cola de notificaciones BullMQ

¿Por qué Socket.io y no WebSocket plano?

Socket.io nos da rooms (agrupar sockets por chat o por usuario).

Fallback a polling — si WebSocket no anda, cae a HTTP long-polling.

Reconexión automática — no hay que implementarla a mano.

Middleware de autenticación — se integra con el JWT de cookie.

Adapter de Redis — para escalar horizontalmente entre instancias.

Todo eso hecho a mano con ws raw lleva mucho más código y testing.

Además, Socket.io tiene connectionStateRecovery que permite recuperar mensajes perdidos durante una desconexión de hasta 2 minutos.

¿Cómo se conecta un usuario?

Paso a paso

  1. 1. El frontend se conecta a Socket.io con { withCredentials: true }.
    Esto envía las cookies automáticamente.
  2. 2. El middleware io.use() intercepta la conexión.
    Extrae tandem_access_token de la cookie, verifica el JWT y si es válido, guarda { id, id_tipo_usuario } en socket.data.user.
  3. 3. El handler io.on('connection') une automáticamente el socket al room user:{id}.
    Este room se usa para notificaciones personalizadas.
  4. 4. El frontend emite chat:join para cada chat que el usuario quiera escuchar.
  5. 5. El servidor verifica que el usuario sea participante activo del chat (contra DB o cache Redis) y une el socket al room chat:{id_chat}.

¿Cómo se decide si el usuario puede enviar?

El server valida dos cosas antes de insertar el mensaje:

1) Que el usuario sea participante activo del chat (contra DB o Redis cache).

2) Si el perteneciente es tutelado y el chat es con un profesional, verifica el permiso EnviarMensajes o ChatearConProfesional según corresponda.

Todo esto pasa antes de INSERTAR el mensaje en la DB.

Redis & Caché

Opcional — ioredis para Socket.io adapter, BullMQ, caché de consultas y rate limiting

¿Por qué Redis es opcional?

En desarrollo no queremos depender de tener Redis instalado.

Si REDIS_URL no está configurado, todo funciona igual:

Socket.io no usa adapter (sirve para 1 instancia).

Las notificaciones se procesan en el mismo proceso sin BullMQ.

El cacheo simplemente no ocurre.

En producción, activás Redis y obtenés:

Escalado horizontal — múltiples instancias comparten estado via Redis adapter.

Colas persistentes — BullMQ no pierde jobs.

Caché de consultas — menos carga en PostgreSQL.

¿Cómo se conecta Redis?

Conexión factory

createRedisConnection() devuelve una instancia de ioredis con nombre descriptivo.

Ej: tandem:socket-io-pub, tandem:bullmq, etc.

Usa lazyConnect: true para no bloquear el startup si Redis no está disponible.

CacheService — patrón cache-aside

getOrSet(key, fetchFn, ttl): si la clave está en Redis la devuelve; si no, ejecuta fetchFn, guarda el resultado y lo devuelve.

delByPattern(pattern): limpia caché por patrón (ej: "permisos:*") usando scanStream.

class CacheService {   constructor(redisClient) {     this.client = redisClient;   }   async getOrSet(key, fetchFn, ttl = 300) {     const cached = await this.get(key);     if (cached !== null) return cached;     const data = await fetchFn();     await this.set(key, data, ttl);     return data;   }   async delByPattern(pattern) {     const stream = this.client.scanStream({ match: pattern });     for await (const keys of stream) {       if (keys.length) await this.client.del(keys);     }   } }

1. ARASAAC — Pictogramas

¿Para qué? Los pictogramas son imágenes que representan conceptos, usadas por personas con TEA para comunicarse.

ARASAAC es el repositorio estándar en español.

¿Cómo funciona?

  1. Búsqueda: el frontend busca pictogramas (GET /api/pictograms/search?q=comer).
  2. Cache: PictogramaService primero revisa la cache de Redis.
    Si está, devuelve directamente.
    Si no, llama a la API de ARASAAC, cachea el resultado y lo devuelve.
  3. Imágenes: cuando se necesita la imagen, se descarga desde static.arasaac.org y se sube a Supabase Storage para tenerla localmente.
  4. Sincronización: un job programado (src/jobs/pictogramaSyncJob.js) sincroniza el catálogo completo periódicamente.

2. DiceBear — Avatares

¿Para qué? Generar avatares estilo ilustración para los perfiles de usuario.

  2. 2. Se descarga el PNG con Axios.
  3. 3. Se sube a Supabase Storage para servirlo como imagen estática.

Permisos

Modelo de autorización por roles con catálogos, defaults por modo y auditoría

¿Por qué este modelo y no CASL o similar?

El modelo de Tándem tiene particularidades:

Los pertenecientes pueden operar en dos modos (autogestionado / tutelado).

Los tutores autorizan sobre múltiples pertenecientes.

Los profesionales tienen permisos específicos por vínculo.

Una librería genérica como CASL requeriría adaptadores que al final hacen lo mismo que AuthorizationService.can().

Además, necesitamos auditoría de cada cambio de permiso (quién, cuándo, valor anterior/nuevo).

Con el modelo actual, el histórico se guarda automáticamente en tablas separadas.

¿Cómo funciona?

1. Roles

Cada usuario tiene un id_tipo_usuario que define su rol:

  • Perteneciente — usa la app
  • Tutor — autoriza permisos
  • Profesional — trabaja con vinculados
  • Admin — herramientas internas

2. Modos del Perteneciente

El campo puede_autogestionarse determina el modo:

  • Autogestionado (true) — permisos default abiertos
  • Tutelado (false) — requiere autorización del tutor

3. Catálogos y defaults

Dos catálogos definen los permisos disponibles: uno para pertenecientes y otro para profesionales.

Cada permiso tiene un default según el modo.

4. AuthorizationService

Service central que recibe (usuario, accion, contexto) y determina si la acción está permitida.

Busca en permisos_otorgados. Si no encuentra, usa el default del catálogo.

Arquitectura en Capas

Express 5 + capa de rutas, controladores, servicios, repositorios y base de datos

¿Por qué esta separación en capas?

Cada capa tiene una responsabilidad única y se puede testear de forma aislada:

  • Controller — solo maneja HTTP (req/res). No tiene lógica de negocio.
  • Service — orquesta la lógica, llama a repositorios, cache, APIs externas.
  • Repository — solo SQL. No sabe de HTTP ni de reglas de negocio.
  • Entity — define la forma de los datos. Sin comportamiento.
